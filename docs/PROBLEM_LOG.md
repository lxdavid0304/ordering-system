# 問題與解法紀錄

本文件記錄已在目前版本處理的實際問題、根因、解法與後續預防措施。新問題應以日期新增，不覆寫既有紀錄。

## 2026-07-20：LINE 狀態通知被舊 job 阻塞

**症狀**：會員可收到「待確認訂金」，但管理者確認訂金後的「採買進行中」、商品買齊後的「待取貨」、尾款完成後的「已完成」沒有即時送達。

**根因**：通知以佇列順序處理。早期 `pending_deposit` job 若仍是 `pending` 或 `processing`，舊工作者會讓同一訂單的後續狀態一直等待。部分流程也過度依賴 RPC 回傳資料的形狀，導致最新狀態未被穩定地傳給通知函式。

**解法**：

- 後台狀態、待取貨與付款流程正規化 RPC 回傳的訂單資料，並明確傳送 `target_status`。
- `line-notify` 在指定最新狀態時，將同一訂單較舊的 `pending`、`failed`、`processing` job 標記為 `skipped`。
- 指定狀態的投遞不再等待過時 job；缺少狀態 job 時由工作者補建快照。
- 顯示單筆訂單的通知工作與系統診斷，讓失敗可被管理者看見。

**驗收**：依序確認訂金、標記待取貨、儲存尾款，LINE 應分別收到 `open`、`ready_pickup`、`fulfilled`。訂單抽屜中的 job 狀態應為 `sent`。

## 2026-07-20：LINE 初始通知可能使用 0 元草稿金額

**症狀**：新訂單建立時，若 INSERT trigger 先於訂單項目與總額計算執行，通知可能帶有 0 元金額。

**根因**：訂單主檔先以 0 元建立，後續才由 RPC 寫入項目並更新總額；INSERT trigger 的時間點過早。

**解法**：移除 INSERT trigger；由 `create-order` 在完成總額計算後建立初始通知快照。工作者會略過既有未送出的 0 元草稿 job。

**預防**：通知內容涉及金額時，必須在最終總額寫入後才建立 job。

## 2026-07-20：延遲重試使用了後續訂單狀態或金額

**症狀**：排隊或重試的通知可能描述較晚的狀態，而不是當初異動時的內容。

**根因**：工作者若只讀取當前 `orders` 資料，訂單已在等待期間改為下一個狀態或被調整金額。

**解法**：狀態 trigger 寫入 `to_status`、交貨地點、報價、實際總額、已收款與價格異動的 JSON 快照；通知文字優先使用 job payload。

**預防**：所有非同步通知都必須保存事件快照，不能以處理時的可變資料取代。

## 2026-07-20：實際採購金額與價格異動未被完整保留

**症狀**：採購後若總額改變，會員無法清楚辨識原報價、實際總額與剩餘尾款。

**根因**：原訂單只保存單一總額，調整後會失去原報價。

**解法**：新增 `quoted_total_amount`、調整時間與調整者；`admin_mark_order_ready_for_pickup` 將採購確認與狀態轉換包成單一 RPC，調整金額必須填原因。LINE 待取貨通知會顯示價格異動。

**預防**：任何金額調整都必須保留原值、原因、操作者與時間。

## 2026-07-20：通知投遞失敗無法可靠重試

**症狀**：LINE API 暫時失敗或函式中斷後，通知可能卡在處理中或被永久略過。

**根因**：通知 job 缺少處理 claim、過期回收與重試排程。

**解法**：新增 `claim_token`、`processing_started_at`、`next_attempt_at`，以有限次數與遞增延遲重試；過期的 processing job 會回到 pending。診斷函式提供 job 狀態與錯誤。

**預防**：任何佇列工作都要有 claim、timeout 回收、重試上限與可觀測性。

## 2026-07-20：既有資料庫缺少最新狀態通知 trigger

**症狀**：只建立初始通知，後續狀態異動沒有對應的通知 job。

**根因**：雲端資料庫尚未套用重新建立 trigger 的 migration，或仍使用只記錄舊欄位的函式版本。

**解法**：`20260720140000_rebuild_line_status_notification_trigger.sql` 重新定義函式並重建 `orders` 的狀態更新 trigger。`line-notify` 同時保有補建指定狀態 job 的容錯能力。

**預防**：每次資料庫行為變更都要以新 migration 部署，並用新訂單走完整狀態流程驗收。

## 2026-07-20：訂單總額、運費與利潤計算不一致

**症狀**：手動修改品項或熱門商品成本後，訂單總額、客戶運費與營運利潤可能不同步。

**根因**：原有總額 trigger 只累計品項小計，沒有把客戶運費與成本運費快照納入。

**解法**：擴充商品與品項成本欄位，更新 `create_order` 與 `update_order_total`，以品項快照計算 `total_amount`、`shipping_amount` 與 `profit_amount`。

**預防**：訂單建立後的商業數值應依訂單品項快照計算，不可回讀可能變動的商品主檔。
