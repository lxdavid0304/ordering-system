## 專案背景

### 專案名稱

- 訂購系統

### 專案目標

- 建立可公開分享連結的訂購系統，取代每週重開 Google 表單的流程
- 讓使用者可直接填寫姓名、電話、送貨地點、備註與多筆商品資料
- 每筆商品支援商品名稱、單價、數量，並自動計算整筆訂單總金額
- 建立管理後台，可依送貨地點彙整訂單並調整每週開放時段
- 串接 Supabase、SQL、RLS 與管理端登入保護
- 保留後續擴充會員、付款方式、待確認訂單等流程的空間

### 使用工具

- Roo Code
- Supabase
- SQL / PostgreSQL
- HTML / CSS / JavaScript
- React
- Chrome DevTools
- Python `http.server`

### Prompt 01：初始系統啟動 / 專案理解

#### 使用時機

- 專案剛開始
- 需要代理先理解整體架構、頁面、流程與資料結構
- 不希望它一開始就直接亂改

#### 原始 Prompt

```text
請先完整理解這個專案，再開始做任何修改。

你要先做的不是寫程式，而是整理以下內容：
1. 專案目前有哪些頁面與功能
```

#### 成效評估

- 很適合專案初期。
- 缺點是如果 Prompt 太寬，模型可能只給很空泛的摘要。
- 改善方式是明確指定要列出檔案、功能、風險與下一步。

### Prompt 01A：訂購系統需求整合 / MVP 對齊

#### 使用時機

- 當專案需求已經從想法變成具體功能，需要一次把前台、後台、資料庫與時段規則講清楚時
- 當希望代理先依照固定需求做 MVP，而不是自行想像產品範圍時

#### 原始 Prompt

```text
我要做一個訂購系統，因為我不想每個禮拜都重新開新的 Google 表單。
這個系統要讓任何知道連結的人都可以直接填單，不用先做會員登入，但之後要能擴充。

需求如下：
1. 前台表單欄位要有姓名、電話、運送地址、商品欄、備註。
2. 運送地址要做成下拉選單，固定只有明德樓、據德樓、蘊德樓、機車停車場。
3. 商品可以同時輸入多筆，每筆包含商品名稱、價格、數量。
4. 單項商品金額由使用者自己填，但系統要依據每筆價格與數量自動計算總金額。
5. 後端資料要能即時存檔，管理端打開後要能看到最新資料，並且依送貨地點幫我整理清單。
6. 時間要用台灣時間，我要能自己設定每週星期幾幾點開放或關閉填單，每個小時一個選項。
7. 前端頁面也要顯示目前開放或關閉狀態。
8. 管理端要有隱藏網址 `/admin-david0304`，也要有密碼保護。
9. 後端資料庫希望能用 SQL，因為後續可能需要整理與查詢資料。

請先依照這些需求幫我規劃並實作 MVP，優先做出可公開填單、可存資料、可在後台查看與調整開放時間的版本。
```

#### Prompt 設計分析

- 這個 Prompt 一次定義了產品目標、固定欄位、地址選項、權限模型與資料庫方向，邊界很清楚。
- 它有效限制代理不要擅自改成會員制、不要把地址改成自由輸入，也不要忽略後台與開放時段。
- 它把 MVP 範圍壓在「公開填單 + 即時存檔 + 後台管理」這條主線上，適合專案初期快速落地。

#### 成效評估

- 對功能對齊很有幫助，因為前後台需求一次講清楚。
- 對實作節奏也有幫助，代理能先收斂到 Supabase + SQL 的方案。
- 不足之處是沒有先明講要如何驗證 key、RLS 與權限設定，後續串接時仍需補一輪除錯 Prompt。

### Prompt 02：Orchestration 規劃

#### 使用時機

- 任務變大，不適合一次做完
- 需要先拆成幾個子問題
- 想讓代理先規劃順序，而不是亂跳

#### 原始 Prompt

```text
請先不要直接修改，先幫我把這個需求拆成一個可執行的 Orchestration 計畫。

請至少整理：
1. 主要目標
2. 需要先確認的阻塞點
3. 可以平行做的子任務
4. 最合理的實作順序
5. 每一步完成後要怎麼驗證

如果有哪一步風險最高，也請標記出來。
```

#### Prompt 設計分析

- 它把任務從「直接做」改成「先規劃再做」。
- 特別適合跨前端、資料庫、Edge Functions 的需求。
- 可以減少代理在大型需求中迷失方向。

#### 成效評估

- 對中大型需求很有效。
- 不適合超小需求，會顯得太重。
- 若之後真的執行，最好再搭配明確的實作 Prompt。

### Prompt 03：拆模組 / 拆任務

#### 使用時機

- 已知目前檔案太大
- 一個頁面把 UI、狀態、API、驗證全部混在一起
- 想請代理協助拆成可維護結構

#### 原始 Prompt

```text
請幫我把這個功能重新整理成可維護的模組結構。

要求：
1. 先分析目前檔案中有哪些責任混在一起
2. 說明哪些應拆到 component / service / util / context
3. 如果要重構，請按照最小風險順序拆開
4. 不要改變原本的功能行為
5. 重構後請說明哪些責任被移到哪裡
```

#### Prompt 設計分析

- 這個 Prompt 不是直接要求「重寫」，而是要求「責任切分」。
- 適合前端頁面過大，或單檔案邏輯過重的情況。
- 也很適合搭配 React 遷移前的盤點。

#### 成效評估

- 能有效降低大型檔案的混亂度。
- 如果沒有要求「不要改變行為」，代理容易順手改功能。
- 需要搭配驗證步驟一起使用。

## Prompt 設計分析總結

### 1. 我在這個專案中的 Prompt 設計策略

- 先要求理解，再要求修改
- 先切邊界，再進行實作
- 優先維持既有可用流程，不做無謂重寫
- 每次都盡量要求代理交代驗證方式

### 2. 哪些 Prompt 明顯提升了品質？

- 初始理解類 Prompt
- Orchestration 規劃類 Prompt
- 權限 / 付款 / 部署這類邊界清楚的 Prompt

### 3. 哪些 Prompt 失敗或導致 Roo Code 走偏？

- 只說「幫我做」但沒有限制邊界的 Prompt
- 沒交代保留哪些部分、哪些不能動的 Prompt
- 沒要求驗證的 Prompt

### 4. Orchestration 是否真的有幫助？

- 對大需求有幫助
- 對小需求容易太重
- 最適合跨前端、後端、資料層的工作

### 5. 如果不用 Orchestration，會卡在哪裡？

- 代理會直接跳進實作
- 容易忽略真正的阻塞點
- 跨層任務容易變成東補一塊、西補一塊

### 6. 我後來如何優化 Prompt？

- 把「情境」寫得更明確
- 把「不能做什麼」寫清楚
- 把「最後要交付什麼」寫清楚
- 把「如何驗證」寫進 Prompt

## Prompt 11：將既有 HTML / CSS / JavaScript 專案逐步遷移到 React

### 適用情境

- 目前專案前端是傳統多頁式 HTML + CSS + JavaScript
- 想改成 React，但不希望直接把現有可運作版本全部打掉
- 後端已存在，例如 Supabase、SQL schema、Edge Functions、既有 API
- 需要先判斷哪些能保留、哪些必須重寫，再開始遷移

### Prompt 範本

```text
請先檢查我目前的前端專案結構，判斷哪些部分可以保留、哪些部分需要改寫成 React。

遷移時請遵守以下原則：
1. 不要先破壞現有可運作的原生 HTML 版本，先建立一套獨立的 React 前端。
2. 優先保留後端、資料庫、Supabase 設定、SQL、Edge Functions、商業規則。
3. 將前端重新整理為 React 的 pages、components、services、utils、context。
4. 先把共用邏輯抽離，再逐頁遷移，不要只做畫面，要把實際流程接回去。
5. 需要保留原本的登入、註冊、訂單填寫、歷史訂單、會員資料、修改密碼、付款流程、待處理訂單、管理後台。
6. 如果現有程式裡有 localStorage、sessionStorage、Supabase auth、表單驗證、狀態切換、訂單送出流程，也要一起遷移。
7. 請把新 React 專案建成可以執行與 build 的狀態，並在完成後驗證一次。
8. 不要直接假設要重寫後端；除非我明確要求，否則後端只做最小必要調整。
9. 最後請清楚告訴我：
   - 哪些是保留沿用
   - 哪些是 React 重寫
   - 新增了哪些檔案
   - 怎麼啟動 React 版本

如果你發現現有程式碼有編碼錯誤、重複模板、DOM 操作過重、邏輯和畫面混雜，也請一起整理並說明。
```

### Prompt 作用說明

- 這個 Prompt 的重點不是直接要求「改成 React」，而是先要求模型判讀結構。
- 它會讓代理先區分 UI 層、商業邏輯、後端資料層。
- 能有效避免把 SQL、Supabase、Edge Functions 一起不必要地重寫。
- 它也明確要求先建立並行版本，降低重構風險。

### 為什麼這個 Prompt 有效

- 它先限制遷移邊界，避免模型把整個專案當成要重做。
- 它指定遷移順序：先盤點、再抽共用邏輯、再逐頁搬。
- 它要求保留實際業務流程，而不是只做靜態 React 畫面。
- 它要求最後一定要 build 驗證，避免只停在「看起來完成」。

### 使用時的注意事項

- 如果專案已有既定技術棧，要補上「延續既有工具」。
- 如果只想先搬某幾頁，要在 Prompt 中明講優先順序。
- 如果後端不能動，要明確寫「後端只讀不改」。
- 如果已經有正式上線版本，應補上「不要覆蓋現有入口，先做並行版本」。

### 常見失誤

- 只說「幫我改成 React」，沒有交代保留與重寫邊界。
- 沒有指定要保留既有商業規則，最後只得到靜態畫面。
- 沒有要求 build 驗證，產出最後不能跑。
- 沒有要求與原版本並行，直接覆蓋現有可用前端。
